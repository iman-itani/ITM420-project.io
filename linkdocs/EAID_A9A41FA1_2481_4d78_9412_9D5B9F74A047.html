Use-Case Name:
Extract Data

Other Participating Actors:
Database (external server)
Description:
The Extract Data use-case involves the process of retrieving information related to the patient, including patient number (ID), date of service, name, and nationality from the database system. The primary system actor involved in this use-case is the user, who initiates the process of extracting data from the external database server. The data extracted will be used for billing and other related purposes.
Precondition:
 The record looked at should exist in the database system.
The user must have a valid login credential to access the system and must be authorized to retrieve patient data.
Trigger:
Other use cases
Typical Course of Events:
System response

 
 Step1: The system prompts the user to enter the primary key- identifier of the needed record.
Step2: The user enters the primary key into the system.
Step3: The system sends a request to the Phenics database server to retrieve needed information.
Step4: The system assigns a date automatically to the transaction.
Step5: The system displays the retrieved information on the screen for the user to review.
Alternate Courses:
 Alternate to step 3:  The primary key entered by the user is invalid.
1. The system displays an error message indicating that the primary key entered is invalid.
The use-case continues from step 2.
Conclusion:
Upon successful completion of the use-case, the user can retrieve patient information from the external database server and use it for billing or other related purposes.
Postcondition:
The information retrieved from the Phenics database server is stored in the system and can be accessed by authorized personnel for future reference.
Business Rules:
 Patient Phenics ID consists of family ID and individual ID.
Only authorized personnel can access patient information.
Implementation Constraints and Specifications:
 The system must have a secure connection to the database server.
The system must have a mechanism to authenticate the user before accessing patient data.
The system must have a backup and recovery mechanism in case of data loss or system failure.
Assumptions:
 The external database server is reliable and accessible.
The patient information stored on the external database server is accurate and up to date.
