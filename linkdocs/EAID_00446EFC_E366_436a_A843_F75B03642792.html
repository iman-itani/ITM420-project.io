Use-Case Name:
Remove service
Use Case Type
Business Requirements þ
Use-Case ID:
Priority:
Source:
Interview 2
Primary System Actor:
Administrator
Description:
Accessing the system's management interface, selecting a medical service to be removed, confirming the removal, and notifying relevant parties within the organization of the removal, while the system updates the list of medical services to reflect the removal.
Precondition:
The administrator must have a valid login credential to access the system and must be authorized to remove a service type.
Trigger:
Remove a service type from the system that is no longer needed or valid.
Typical Course Of Events:
Actor
System response

 
 Step 1: The administrator accesses the system's management interface and navigates to the section to remove a service.

Step 3: The administrator selects the medical service to be removed from the list.
Step 5: The administrator confirms the removal of the medical service.
 
 Step 2: The interface presents a list of all available medical services using Extract Data use case.
Step 4: The system prompts the administrator to confirm the removal of the medical service.
Step 6: The system removes the medical service from the list of available services.
Step 7: The system updates the list of medical services to reflect the removal of the service
Step 8: the system notify the cashier of the changes made in the services list.
Alternate Courses:
 Alt to step 3: The service type selected by the administrator does not exist.
  • The system displays an error message indicating that the service type does not exist.
  • The administrator selects a different service type.
  • The use-case continues from step 5.
Alt to step 5: The administrator cancels the removal request.
  • The system cancels the removal request and returns to the previous screen.
 
Conclusion:
Upon successful completion of the use-case, the selected service type is removed from the system, which can no longer be used by the cashier while recording the patient's transactions.
Postcondition:
The selected service type can no longer be used for future transaction
Business Rules:
 The system must have a mechanism to authenticate the user before removing a service type.
Only authorized personnel should be able to remove a service type.
Implementation Constraints and Specifications:
The system must have a backup and recovery mechanism in case of data loss or system failure.
Assumptions:
The details entered by the administrator are accurate and complete.
